## Module-by-module extracted value

### Module 1
- Mindset: AI output is plausible draft code, not verified runtime truth.
- Working loop: ask -> inspect -> run -> test -> refine.
- Output expectation: user stories, ADR paragraph, running `/health`, and reflection evidence.

### Module 2
- Backend progression is staged: strict models -> storage -> CRUD -> business rules -> tests.
- File-aware editor prompting is mandatory for reducing hallucinated project assumptions.
- Break Test is required to prove tests detect real faults.

### Module 3
- Frontend must be verified as behavior, not only visuals.
- Drag-and-drop and fetch handling require explicit event/status logic.
- Debugging evidence should include network status, response bodies, and test outcomes.

### Module 4
- Repo-level AI usage increases blast radius; every generated artifact needs evidence.
- CI and Docker are correctness surfaces, not setup checkboxes.
- Project memory and instruction files (`CLAUDE.md`) shape safer repo-wide execution.

### Module 5
- Governance posture: AI proposes, developer grades.
- Strong guardrails require scope boundaries and file-cited claims.
- Security and context engineering readings support repeatable team-level standards.

### Mid-course + final project
- Mid-course: two scoped end-to-end improvements with full AI workflow evidence.
- Final project: release-readiness proof (CI, Docker, docs, ownership) with no feature creep.
